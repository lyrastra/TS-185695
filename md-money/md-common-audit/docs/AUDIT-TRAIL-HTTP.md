# Использование AuditTrail в HTTP

## Обзор

Система AuditTrail обеспечивает сквозное отслеживание HTTP запросов через распределенную архитектуру микросервисов. Для этого используются специальные HTTP заголовки, которые передают контекст трейсинга между сервисами.

## HTTP заголовки для трейсинга

### 1. `__MdAuditParentScopeContext` (входящий)

**Назначение**: Передача контекста родительского span'а для создания иерархии трейсинга

**Направление**: Входящий запрос → Сервис

**Формат**: JSON-строка

**Структура данных**:
```json
{
  "AsyncTraceId": "guid",
  "TraceId": "guid", 
  "ParentId": "guid?",
  "CurrentId": "guid",
  "CurrentDepth": "short"
}
```

**Описание полей**:
- `AsyncTraceId` (Guid) - идентификатор асинхронного трейса
- `TraceId` (Guid) - идентификатор трейса
- `ParentId` (Guid?) - идентификатор родительского span'а (может быть null)
- `CurrentId` (Guid) - идентификатор текущего span'а
- `CurrentDepth` (short) - глубина вложенности span'а

**Обработка**: Middleware автоматически читает этот заголовок и создает дочерний span относительно родительского контекста.

### 2. `MD-AuditTrail-Context` (исходящий)

**Назначение**: Передача контекста audit trail в ответе для клиентских приложений и внешних систем

**Направление**: Сервис → Исходящий ответ

**Важно**: Этот заголовок носит справочный характер и не критичен для работы системы AuditTrail

**Формат**: Строка с разделителями

**Структура**: `"00;{дата};{AsyncTraceId};{TraceId};{CurrentId}"`

**Описание компонентов**:
- `00` - версия формата заголовка
- `{дата}` - дата в формате `yyyyMMdd` (UTC)
- `{AsyncTraceId}` - идентификатор асинхронного трейса
- `{TraceId}` - идентификатор трейса
- `{CurrentId}` - идентификатор текущего span'а

**Пример**: `"00;20241215;12345678-1234-1234-1234-123456789abc;87654321-4321-4321-4321-cba987654321;11111111-2222-3333-4444-555555555555"`

**Применения**:
- **Клиентская аналитика**: Frontend может отслеживать производительность API
- **Отладка**: Передача TraceId в поддержку для быстрого поиска логов
- **Корреляция событий**: Связывание клиентских событий с серверными логами
- **Compliance**: Соответствие требованиям аудита и трассируемости
- **Мониторинг**: Интеграция с системами мониторинга и APM

## Как работает трейсинг HTTP запросов

### Создание трейса

1. **Первый сервис** в цепочке создает новый span без родительского контекста
2. **Последующие сервисы** получают контекст через заголовок `__MdAuditParentScopeContext`
3. **Middleware** автоматически создает дочерний span относительно родительского

### Передача контекста между сервисами

1. **Обработка запроса**: Middleware читает входящий заголовок `__MdAuditParentScopeContext`
2. **Создание span'а**: Создается дочерний span с контекстом родителя
3. **Обогащение ответа**: В ответ добавляется заголовок `MD-AuditTrail-Context` с данными о текущем спане
4. **Следующий сервис**: Получает контекст и продолжает цепочку трейсинга

## Автоматическая работа системы

### Внутренние системы

После подключения AuditTrail (см. раздел "Подключение AuditTrail для веб-приложений") система **автоматически**:

- **Передает** заголовок `__MdAuditParentScopeContext` в следующий запрос для построения цепочки трейсинга
- **Добавляет** заголовок `MD-AuditTrail-Context` в ответ (для клиентских приложений)

**Важно**: 
- Заголовок `__MdAuditParentScopeContext` **автоматически передается** только между внутренними системами
- Внешние клиенты и системы **не передают** этот заголовок
- Заголовок `MD-AuditTrail-Context` в ответе носит **справочный характер** и не критичен для работы

### Сценарии использования

#### Внутренние микросервисы
```
Service A → Service B → Service C
    ↓           ↓           ↓
Создает span  Получает    Получает
              контекст    контекст
    ↑           ↑           ↑
Отправляет   Отправляет   Отправляет
контекст     контекст     контекст
```

#### Внешние клиенты
```
External Client → Service A → Service B
       ↓              ↓           ↓
   НЕ передает    Создает span  Получает
   заголовок                    контекст
       ↑              ↑           ↑
   Может читать   Отправляет   Отправляет
   контекст       контекст     контекст
```

**Важно**: Внешние клиенты **не должны** передавать заголовок `__MdAuditParentScopeContext`


## Подключение AuditTrail для веб-приложений

### .NET 8+ (ASP.NET Core)

**Middleware**: `app.UseAuditApiHandlerTrace()`

**Зависимости**: 
- `Moedelo.Common.Audit.Middleware`
- `Moedelo.Common.Audit`

### .NET Framework 4.7.2

**WebAPI приложения**:
```csharp
// Global.asax.cs
GlobalConfiguration.Configuration.Filters.Add(ServiceInstaller.GetInstance<AuditTrailWebApiFilter>());
```

**MVC приложения**:
```csharp
// Global.asax.cs
GlobalFilters.Filters.Add(ServiceInstaller.GetInstance<MvcAuditFilter>());
```

**Зависимости** (v2 инфраструктура):
- `Moedelo.CommonV2.Audit`
- `Moedelo.CommonV2.Audit.Implementations`
- `Moedelo.InfrastructureV2.Audit`
- `Moedelo.InfrastructureV2.RedisDataAccess`


## Производительность

### Влияние на производительность

- **Размер заголовков**: Каждый заголовок добавляет ~200-500 байт к HTTP запросу/ответу
- **Парсинг**: JSON парсинг добавляет ~0.1-1ms на запрос
- **Сетевой трафик**: Минимальное увеличение размера пакетов

На практике значимого влияния подключения auditTrail для http запросов на производительность не выялено
